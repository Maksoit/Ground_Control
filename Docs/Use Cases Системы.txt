Use Cases Системы
UC-01 — Управление рейсом (сквозной сценарий: от продажи до вылета)
Цель: провести рейс по стадиям аэропорта: продажа → регистрация → обслуживание → посадка → вылет.
Акторы:
* Пассажир (виртуальный, генерируемый или ручной)

* Оператор кассы (через UI)
ку
* Оператор регистрации (через UI)

* Супервайзер наземного обслуживания (через UI)

* Диспетчер движения (через UI)

* Наземная техника (как сервисы-исполнители)

* Самолёт (как сервис состояния борта)

* Сервис рейсов (источник расписания)

Предусловия:
   * В системе есть рейсы с расписанием.

   * Есть самолёт, назначенный на рейс.

   * Работает единый фронт (или хотя бы страницы модулей).

   * Kafka доступна для событий.

Основной поток:
      1. В системе существует рейс со статусом “по расписанию”.

      2. Пассажир создаётся и выбирает рейс.

      3. Через кассу покупается билет, место уменьшается.

      4. В момент открытия регистрации рейс меняет стадию на “регистрация открыта”.

      5. Регистрация принимает пассажиров, проверяет билеты, фиксирует багаж и питание.

      6. Зарегистрированные пассажиры попадают в накопитель ожидания.

      7. Супервайзер запускает цепочку обслуживания самолёта.

      8. Техника выполняет операции (топливо/питание/багаж/подвоз пассажиров).

      9. После завершения операций самолёт становится “готов к вылету”.

      10. Самолёт запрашивает разрешение на руление.

      11. Диспетчер проверяет занятость дорог и выдаёт разрешение.

      12. Самолёт выезжает и рейс переходит в состояние “вылетел”.

      13. Табло/фронт отображает обновлённый статус.

Постусловия:
         * Рейс имеет статус “вылетел”.

         * Пассажиры имеют статус “на борту”.

         * Все операции залогированы.























UC-02 — Возврат билета
Цель: пассажир возвращает билет, место освобождается.
Основной поток:
            1. Оператор/пассажир выбирает билет.

            2. Отправляет запрос на возврат.

            3. Касса проверяет допустимость возврата (например, до закрытия регистрации).

            4. Статус билета становится “возвращён”.

            5. Место на рейсе освобождается.

            6. Фронт показывает обновление.
UC-03 — Регистрация пассажира
Цель: зарегистрировать пассажира на рейс.
Основной поток:
               1. Регистрация получает список билетов или умеет проверять билет онлайн.

               2. Оператор вводит билет/ID пассажира.

               3. Сервис проверяет: рейс активен, регистрация открыта, билет действителен.

               4. Фиксируется багаж.

               5. Пассажир становится зарегистрированным.

Альтернативы:
                  * билет возвращён → отказ

                  * регистрация закрыта → отказ

                  * пассажир опоздал → отказ/статус missed flight
UC-04 — Управление наземным обслуживанием
Цель: обеспечить выполнение всех операций обслуживания самолёта до готовности.
Основной поток:
                     1. Супервайзер видит самолёт/рейс и список требуемых операций.

                     2. Выдаёт задачи технике.

                     3. Получает отчёты о выполнении.

                     4. Если всё выполнено — самолёт готов.

                     5. Если нет — фиксируется причина задержки.

UC-05 — Разрешение на руление
Цель: безопасно вывести самолёт на маршрут.
Основной поток:
                        1. Самолёт запрашивает разрешение на выезд.

                        2. Диспетчер проверяет занятость дорог.

                        3. Если свободно — выдаёт разрешение.

                        4. Если занято — самолёт ждёт.

                        5. После освобождения — разрешение выдаётся.
